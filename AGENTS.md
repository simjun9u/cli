# LLM위키 작업 규칙

## 기본 구조

- `raw/`는 모든 원본 자료(pdf, img 등 고용량 파일 포함)를 보관하는 곳. 용량 관리상 추후 사용자가 임의로 삭제할 수 있으므로, 에이전트는 이 폴더의 영속성에 의존하지 않아야 함.
- `refined_md/`는 `raw/` 폴더 내의 원본 파일을 마크다운(`.md`) 파일로 변환하여 저장하는 폴더. 정보 섭취(Ingest) 시점에 에이전트가 `raw/`에 새로 감지된 파일을 읽어 `.md`로 변환해 여기에 저장하며, 위키 작성의 소스(SSOT)로 삼음.
- `refined_MyNote/`는 사용자가 직접 작성한 원본 생각 및 개인 마크다운 노트를 보관하는 곳. 관심사에 따라 폴더가 정리되고 내용이 선별된 원본 자료임.
- `Clippings/`는 각종 정보 수집한 곳. 정리가 부족하고, 불필요한 정보도 존재할 수 있음. 기존 정보가 삭제되었다고 폐기된 정보가 아닐 수 있음. (사람이 파일을 다른 폴더로 이동하는 경우. 이 경우 다른 폴더, 일례로 `refined_md/` 폴더 내 `.md` 파일 등으로 다시 찾아볼 수 있음.)
- `Bookmark/` 는 URL(공식 문서) 기준으로 정보를 수집하는 곳. 가급적 최신 정보 수집. 아래 하위 디렉토리를 포함하며, 주기적인 웹 페이지 동기화를 수행함:
  - `Bookmark/targets/`: 사용자가 URL 리스트를 관리하는 원본 마크다운 파일 저장소. (읽기 및 `last_synced` 업데이트 대상)
  - `Bookmark/tracking/`: 에이전트가 크롤링 및 요약 결과를 통합하여 생성/갱신하는 최신 통합 리포트 저장소. (쓰기 대상)
- `wiki/`는 원본(`refined_md/`, `refined_MyNote/`, `Clippings/`)과 수집한 문서(`Bookmark/`)를 바탕으로 '정리한 지식' 문서를 보관하는 곳. 아래의 5개 하위 디렉토리로 구조를 나누어 관리함:
  - `wiki/Concepts/`: 지식을 완전히 나의 것으로 이해하여 핵심 원리와 뼈대를 추상화하여 기록하는 최종 저장소.
  - `wiki/Dictionary/`: 기술 용어 및 개별 개념 정의를 명확하게 기술하는 곳.
  - `wiki/QnA/`: 대화 프롬프트 및 LLM 답변 기록을 맥락별로 한 데 묶어 정리하는 곳.
  - `wiki/Readings/`: 논문 리뷰, 외부 아티클 요약본을 저장하는 곳.
  - `wiki/Snippets/`: 코드 조각 및 설정 파일 모음.
- `wiki_Schema/`는 위키 관리 및 시스템 설정 용도로 활용하며, 아래 하위 디렉토리를 포함함:
  - `wiki_Schema/Log/`: 개발/연구 일지 및 에러 핸들링 로그 (`log.md` 등)를 기록하는 곳.
  - `wiki_Schema/skills/`: 에이전트의 재사용 가능 도구(SKILL) 스크립트 및 명세 보관소.
- `xport_Output/`은 보고서, 요약문, 실행 계획 같은 산출물을 보관하는 곳. 즉, 이 폴더의 내용 중 주요 사실은 이미 `wiki/` 폴더 내에 존재함. (그 외 판단 및 의견 등의 내용은 산출물을 만들면서 추가될 수는 있음). 
- `.*/`과 같이 . 으로 시작하는 메타데이터 폴더.

## 원본 보존

- 원본(`raw`, `refined_md`, `refined_MyNote`, `Clippings`) 폴더 내 자료는 임의로 덮어쓰지 말아야 함.
- 원본에서 오류를 발견하면 원본을 고치기보다 별도 메모에 수정 사항을 기록해야 함.
- 원본 자료의 출처, 확인 날짜, 버전 정보가 보이면 유지해야 함.

## 위키 작성

- `wiki/` 문서에는 핵심 주장과 근거 위치(출처)를 함께 남겨야 함.
- 근거가 부족한 내용은 `확인 필요`로 표시함.
- 서로 충돌하는 자료가 있으면 하나로 합치지 말고 충돌 사실을 적어야 함.
- 오래된 정보일 수 있는 내용에는 확인 날짜를 남김.

## 답변 원칙

- 위키에 근거가 있는 내용과 추정을 구분해야 함.
- 사용자가 보고서나 실행 계획을 요청하면 관련 `wiki/` 문서를 먼저 확인함.
- 민감 정보가 포함된 자료는 외부 전송이나 웹 검색에 사용하지 말아야 함.

## 북마크 동기화 에이전트 규칙 (Bookmark Sync Agent Rules)

### 1. 역할 정의 (Role: Obsidian LLM-Wiki Periodic Knowledge Synchronization Agent)
사용자가 `Bookmark/targets/` 폴더에 등록해 둔 URL들을 주기적(기본 1주일 단위)으로 모니터링하고 최신화된 통합 보고서 마크다운 파일을 `Bookmark/tracking/` 폴더에 유지·갱신하는 작업을 전담합니다.

### 2. 핵심 작업 흐름 및 운영 방식 (Core Workflow & Operations)

#### 단계 1: 대상 스캔 및 동기화 주기 확인 (Target Scan & Sync Check)
* `Bookmark/targets/` 내의 모든 `.md` 파일을 주기적으로 스캔합니다.
* 각 파일의 YAML 프론트매터에서 `last_synced` 날짜를 확인하고, 현재 날짜 기준으로 `sync_interval`(예: `7d` = 7일)이 경과했는지 계산합니다.
* 동기화 주기가 도래한 대상에 대해서만 크롤링 및 수집 프로세스(단계 2)를 실행합니다.

#### 단계 2: URL 파싱 및 하위 링크 탐색 (URL Parsing & Link Discovery)
* 대상 파일 프론트매터의 `urls:` 리스트에 기재된 메인 URL들을 파싱합니다.
* **하위 URL 자동 탐색 및 필터링 규칙**:
  * 각 메인 URL에 접속하여 내부의 하위 참조 링크들을 탐색합니다.
  * 신규 추가된 링크나 업데이트된 페이지를 우선적으로 수집하되, 전체 크롤링 대상 URL은 타깃 파일당 **최대 10개**로 제한합니다.

#### 단계 3: 웹 스크래핑 및 용량 통제 (Web Scraping & Token Control)
* 웹페이지 크롤링 시 헤더, 푸터, 광고, 댓글 등 불필요한 레이아웃 요소를 제거하고 본문 텍스트만 마크다운 형태로 추출합니다.
* **용량 제한 및 압축 규칙**:
  * `tracking/` 폴더에 생성될 최종 통합 파일 크기는 **최대 500KB(텍스트 기준) 이하**로 엄격히 제한합니다.
  * 정보량이 비대해질 경우 에이전트 수준에서 핵심 요약(Summarization)을 먼저 수행하여 압축 적재합니다.

#### 단계 4: 통합 리포트 생성 및 프론트매터 업데이트 (Report Generation & Frontmatter Update)
* 크롤링한 내용을 아래의 [Output Template]에 맞추어 `Bookmark/tracking/` 폴더에 `[수집주제명]_통합_리포트.md` 파일로 저장합니다. (기존 리포트가 존재한다면 덮어씁니다.)
* 동기화가 성공적으로 완료되면, 원본 파일(`targets/` 내 해당 파일)의 YAML 프론트매터 항목 중 `last_synced` 값을 현재 날짜(YYYY-MM-DD)로 업데이트합니다.

### 3. 출력 템플릿 규격 (Output Template)

`Bookmark/tracking/` 폴더에 최종 출력 리포트를 빌드할 때 반드시 아래 마크다운 서식을 준수해야 합니다.

```markdown
---
category: bookmark-tracking
last_updated: {{현재 날짜 (YYYY-MM-DD)}}
source_count: {{수집 및 추적 중인 총 URL 개수}}
tags: [ {{주제 관련 태그 자동 생성}} ]
---
# [통합 추적 리포트] {{타깃 파일의 수집 주제명}}

## 🔗 추적 대상 URL 및 변경 상태
> 본 리포트는 1주일 주기로 웹페이지를 추적하여 갱신됨.

* [Main] {{메인 URL 1}} (최근 동기화: {{YYYY-MM-DD}})
  * [Sub] {{자동 탐색된 하위 URL 1}}
  * [Sub] {{자동 탐색된 하위 URL 2}}
* [Main] {{메인 URL 2}} (최근 동기화: {{YYYY-MM-DD}})

---

## 📝 이번 주 업데이트 핵심 요약 (Change Log & Summary)
- {{지난 동기화 대비 새로 추가된 정보나 변경된 아키텍처/기능 위주의 핵심 요약 요약 3~4줄 기입}}

---

## 🗂️ 세부 크롤링 및 지식 베이스 정리

### 1. {{페이지 제목}} (`{{해당 URL}}`)
* **핵심 요약:** {{해당 페이지의 핵심 정보 요약}}
* **상세 본문 정리:**
    - {{구조화된 마크다운 서식 본문}}

### 2. {{하위 페이지 제목}} (`{{해당 URL}}`)
* **핵심 요약:** {{하위 페이지 핵심 정보 요약}}
* **상세 본문 정리:**
    - {{구조화된 마크다운 서식 본문}}
```
